思路打开了！这确实是最快验证官方例程（`pwm_capture.txt`）是否正常工作的方法——既然代码不动，那就让外部信号去适应它。

在你的系统时钟已经修正为正确的 **168MHz**（此时 `TIM4CLK = 84MHz`） 且 **不修改任何定时器预分频配置** 的前提下，适合这个官方例程输入的“黄金频段”是：**$2\text{kHz} \sim 50\text{kHz}$**。

我们可以算出具体范围：

---

### 1. 信号频率的下限（绝对不能低于这个值）

因为 TIM4 是 16 位定时器，最大只能数到 **65535**。在 84MHz 的数数速度下，定时器不溢出所能维持的最长信号周期为：


$$\text{最大周期} = \frac{65535}{84,000,000\text{Hz}} \approx 0.00078\text{秒} = 780\mu\text{s}$$

将周期换算成频率：


$$\text{最低频率} = \frac{84,000,000}{65535} \approx \mathbf{1281.76\text{ Hz}}$$

> **结论**：输入的波形频率**必须大于 1282Hz**。你之前输入的 100Hz 远低于这个极限，所以必然爆表。

### 2. 信号频率的上限（建议不要高于这个值）

理论上可以测到几百 kHz，但如果频率太高，定时器在一个周期内数不了几选下，分辨率就会变差。
例如，如果输入 **840kHz** 的信号，一个周期定时器只能数 100 下，占空比的精度就只有 1% 了。为了保证占空比测得准，建议频率**不要超过 50kHz**。

---

### 🎯 最推荐的测试信号：10kHz / 占空比 60%

如果你把信号发生器调成 **`10kHz`**，那么在不改动官方例程的情况下：

* **1个周期的总计数（IC2Value）** 应为：$84,000,000 / 10,000 = \mathbf{8400}$。


* **高电平的计数（TIM_GetCapture1）** 应为：$8400 \times 60\% = \mathbf{5040}$。



这样既不会溢出（$8400 < 65535$），又能保证极高的测量精度。

---

### ⚠️ 动一处波形，但代码里必须补回一个符号！

虽然你不打算改定时器配置，但看一眼你发出来的 `stm32f4xx_it.c` 代码，里面这一行**必须**要把注释掉的 `* 100` 恢复，或者手工改一下：

```c
// 原始代码你注释掉了 * 100：
// DutyCycle = (TIM_GetCapture1(TIM4) ) / IC2Value;

// 必须改成：
DutyCycle = (TIM_GetCapture1(TIM4) * 100) / IC2Value;

```

**为什么？**
如果不加 `* 100`，当输入 10kHz、60% 占空比时，C 语言在做整数除法：


$$\text{DutyCycle} = \frac{5040}{8400} = 0.6$$


因为没有乘以 100 且是整型变量，`0.6` 会被直接截断取整，你最后在变量里看到的**占空比永远是 0**。

**总结**：把波形改成 **10kHz**，把中断里计算占空比的 **`* 100`** 恢复，你就能立刻看到完全正确的 `Frequency = 10000` 和 `DutyCycle = 60` 了！

理解这个“黄金频段”的原理，其实就是拆解定时器在硬件上的**两个极限**：一个是**硬件容纳的极限（决定下限）**，另一个是**测量精度与 CPU 承受力的极限（决定上限）**。

在不修改任何预分频配置的前提下，TIM4 的计数时钟（TIM4CLK）由于主频配置正确，正全速运行在 **84MHz**。这意味着定时器每秒钟会数 84,000,000 下，每数一下耗时约 **11.9 纳秒 (ns)**。

以此为基础，我们来看看这上下限是如何被定死出来的：

---

## 🧱 下限：为什么不能低于 ~2kHz？（硬件死线）

这完全是由 TIM4 的**寄存器位数**决定的。STM32F407 的 TIM4 是一个 **16位定时器**，它的计数器（CNT）最大只能从 0 数到 $2^{16}-1$，也就是 **65535**。

如果计数器数满了 65535 指针就会清零（溢出），重新从 0 开始数。

* **算算时间极限**：定时器全速（84MHz）不停地数，数满 65535 个数需要多久？

$$\text{最大不溢出时间} = \frac{65535}{84,000,000\text{ Hz}} \approx 0.00078\text{ 秒} = 780.18\mu\text{s}$$


* **换算成频率下限**：如果一个信号的周期超过了 $780.18\mu\text{s}$，定时器就会在一周期内发生溢出。把这个时间换算成频率：



$$\text{最低可测频率} = \frac{1}{780.18\mu\text{s}} \approx \mathbf{1281.76\text{ Hz}}$$



> **结论**：**1282Hz 是硬件的绝对死线**。任何低于 1282Hz 的信号（比如你之前输入的 100Hz），定时器在一周期内都会数爆。为了留出一小点安全余量，通常把 **2kHz** 作为不加分频时的推荐低频极限。
> 
> 

---

## 🎯 上限：为什么建议低于 50kHz？（精度与 CPU 的平衡）

频率越高，信号周期就越短，定时器在一个周期里能数出来的“格子”就越少。虽然硬件上它能捕捉到高达几兆赫兹（MHz）的信号，但有两个现实问题卡住了它：

### 1. 测量精度的考量（分辨率）

官方例程计算占空比的公式是 `(TIM_GetCapture1 * 100) / IC2Value`。这里的 `IC2Value` 是一个周期内定时器总共数的次数。

* 如果输入 **50kHz** 的信号，周期是 $20\mu\text{s}$。定时器总共能数：

$$84,000,000\text{ Hz} \times 20\mu\text{s} = \mathbf{1680\text{ 下}}$$



数 1680 下意味着什么？占空比的测量分辨率可以达到 $\frac{1}{1680} \times 100\% \approx 0.06\%$。这已经非常精准了。
* 但如果你把频率提到 **840kHz**，周期变短，定时器总共只能数 **100 下**。
这时候占空比的分辨率就变成了 $\frac{1}{100} \times 100\% = 1\%$。也就是说你只能测出 59%、60%、61%，中间的微小变化全部丢失。

### 2. CPU 中断的开销

在官方例程中，每来一个波形周期，就会触发一次 `TIM4_IRQHandler` 中断。

* 如果信号是 **50kHz**，意味着 CPU 每秒钟要进出中断 50000 次，即每 $20\mu\text{s}$ 进一次。在 168MHz 的主频下，两次中断之间 CPU 还能执行 3360 条指令，应付中断里的那几行计算绰绰有余。


* 如果信号飙到 **1MHz**，CPU 每 $1\mu\text{s}$ 就要进一次中断，两次中断间只有 168 个时钟周期。CPU 刚把中断里的寄存器压栈、执行完计算、弹栈出来，下一个中断又砸过来了。整个 MCU 的算力会被完全卡死在输入捕获中断里，根本没办法去运行 `main()` 函数里的其他代码。

## 📊 总结：黄金频段的本质

| 频率范围 | 定时器表现 | 结果 | 原因 |
| --- | --- | --- | --- |
| **< 1.28kHz** | 计数器数爆（超过 65535） | ❌ 测出诡异的错误大数值

 | 16位计数器硬件溢出

 |
| **2kHz ~ 50kHz** | 计数在一周期内落在 1680 ~ 42000 之间 | **保持极高精度，CPU很轻松** | **最佳平衡点（黄金频段）** |
| **> 500kHz** | 一周期内数不了几十下，且中断极快 | ❌ 精度严重丢失，系统可能卡死

 | 采样率不足 & 中断开销过大

 |

所以，官方写这个不加分频的 Example，默认就是在测试高频且精准的小信号。你给它塞一个 100Hz 的大波浪，它小小的 16 位容器确实是装不下的。
