思路打开了！这确实是最快验证官方例程（`pwm_capture.txt`）是否正常工作的方法——既然代码不动，那就让外部信号去适应它。

在你的系统时钟已经修正为正确的 **168MHz**（此时 `TIM4CLK = 84MHz`） 且 **不修改任何定时器预分频配置** 的前提下，适合这个官方例程输入的“黄金频段”是：**$2\text{kHz} \sim 50\text{kHz}$**。

我们可以算出具体范围：

---

### 1. 信号频率的下限（绝对不能低于这个值）

因为 TIM4 是 16 位定时器，最大只能数到 **65535**。在 84MHz 的数数速度下，定时器不溢出所能维持的最长信号周期为：


$$\text{最大周期} = \frac{65535}{84,000,000\text{Hz}} \approx 0.00078\text{秒} = 780\mu\text{s}$$

将周期换算成频率：


$$\text{最低频率} = \frac{84,000,000}{65535} \approx \mathbf{1281.76\text{ Hz}}$$

> **结论**：输入的波形频率**必须大于 1282Hz**。你之前输入的 100Hz 远低于这个极限，所以必然爆表。

### 2. 信号频率的上限（建议不要高于这个值）

理论上可以测到几百 kHz，但如果频率太高，定时器在一个周期内数不了几选下，分辨率就会变差。
例如，如果输入 **840kHz** 的信号，一个周期定时器只能数 100 下，占空比的精度就只有 1% 了。为了保证占空比测得准，建议频率**不要超过 50kHz**。

---

### 🎯 最推荐的测试信号：10kHz / 占空比 60%

如果你把信号发生器调成 **`10kHz`**，那么在不改动官方例程的情况下：

* **1个周期的总计数（IC2Value）** 应为：$84,000,000 / 10,000 = \mathbf{8400}$。


* **高电平的计数（TIM_GetCapture1）** 应为：$8400 \times 60\% = \mathbf{5040}$。



这样既不会溢出（$8400 < 65535$），又能保证极高的测量精度。

---

### ⚠️ 动一处波形，但代码里必须补回一个符号！

虽然你不打算改定时器配置，但看一眼你发出来的 `stm32f4xx_it.c` 代码，里面这一行**必须**要把注释掉的 `* 100` 恢复，或者手工改一下：

```c
// 原始代码你注释掉了 * 100：
// DutyCycle = (TIM_GetCapture1(TIM4) ) / IC2Value;

// 必须改成：
DutyCycle = (TIM_GetCapture1(TIM4) * 100) / IC2Value;

```

**为什么？**
如果不加 `* 100`，当输入 10kHz、60% 占空比时，C 语言在做整数除法：


$$\text{DutyCycle} = \frac{5040}{8400} = 0.6$$


因为没有乘以 100 且是整型变量，`0.6` 会被直接截断取整，你最后在变量里看到的**占空比永远是 0**。

**总结**：把波形改成 **10kHz**，把中断里计算占空比的 **`* 100`** 恢复，你就能立刻看到完全正确的 `Frequency = 10000` 和 `DutyCycle = 60` 了！

理解这个“黄金频段”的原理，其实就是拆解定时器在硬件上的**两个极限**：一个是**硬件容纳的极限（决定下限）**，另一个是**测量精度与 CPU 承受力的极限（决定上限）**。

在不修改任何预分频配置的前提下，TIM4 的计数时钟（TIM4CLK）由于主频配置正确，正全速运行在 **84MHz**。这意味着定时器每秒钟会数 84,000,000 下，每数一下耗时约 **11.9 纳秒 (ns)**。

以此为基础，我们来看看这上下限是如何被定死出来的：

---

## 🧱 下限：为什么不能低于 ~2kHz？（硬件死线）

这完全是由 TIM4 的**寄存器位数**决定的。STM32F407 的 TIM4 是一个 **16位定时器**，它的计数器（CNT）最大只能从 0 数到 $2^{16}-1$，也就是 **65535**。

如果计数器数满了 65535 指针就会清零（溢出），重新从 0 开始数。

* **算算时间极限**：定时器全速（84MHz）不停地数，数满 65535 个数需要多久？

$$\text{最大不溢出时间} = \frac{65535}{84,000,000\text{ Hz}} \approx 0.00078\text{ 秒} = 780.18\mu\text{s}$$


* **换算成频率下限**：如果一个信号的周期超过了 $780.18\mu\text{s}$，定时器就会在一周期内发生溢出。把这个时间换算成频率：



$$\text{最低可测频率} = \frac{1}{780.18\mu\text{s}} \approx \mathbf{1281.76\text{ Hz}}$$



> **结论**：**1282Hz 是硬件的绝对死线**。任何低于 1282Hz 的信号（比如你之前输入的 100Hz），定时器在一周期内都会数爆。为了留出一小点安全余量，通常把 **2kHz** 作为不加分频时的推荐低频极限。
> 
> 

---

## 🎯 上限：为什么建议低于 50kHz？（精度与 CPU 的平衡）

频率越高，信号周期就越短，定时器在一个周期里能数出来的“格子”就越少。虽然硬件上它能捕捉到高达几兆赫兹（MHz）的信号，但有两个现实问题卡住了它：

### 1. 测量精度的考量（分辨率）

官方例程计算占空比的公式是 `(TIM_GetCapture1 * 100) / IC2Value`。这里的 `IC2Value` 是一个周期内定时器总共数的次数。

* 如果输入 **50kHz** 的信号，周期是 $20\mu\text{s}$。定时器总共能数：

$$84,000,000\text{ Hz} \times 20\mu\text{s} = \mathbf{1680\text{ 下}}$$



数 1680 下意味着什么？占空比的测量分辨率可以达到 $\frac{1}{1680} \times 100\% \approx 0.06\%$。这已经非常精准了。
* 但如果你把频率提到 **840kHz**，周期变短，定时器总共只能数 **100 下**。
这时候占空比的分辨率就变成了 $\frac{1}{100} \times 100\% = 1\%$。也就是说你只能测出 59%、60%、61%，中间的微小变化全部丢失。

### 2. CPU 中断的开销

在官方例程中，每来一个波形周期，就会触发一次 `TIM4_IRQHandler` 中断。

* 如果信号是 **50kHz**，意味着 CPU 每秒钟要进出中断 50000 次，即每 $20\mu\text{s}$ 进一次。在 168MHz 的主频下，两次中断之间 CPU 还能执行 3360 条指令，应付中断里的那几行计算绰绰有余。


* 如果信号飙到 **1MHz**，CPU 每 $1\mu\text{s}$ 就要进一次中断，两次中断间只有 168 个时钟周期。CPU 刚把中断里的寄存器压栈、执行完计算、弹栈出来，下一个中断又砸过来了。整个 MCU 的算力会被完全卡死在输入捕获中断里，根本没办法去运行 `main()` 函数里的其他代码。

---

## 📊 总结：黄金频段的本质

| 频率范围 | 定时器表现 | 结果 | 原因 |
| --- | --- | --- | --- |
| **< 1.28kHz** | 计数器数爆（超过 65535） | ❌ 测出诡异的错误大数值

 | 16位计数器硬件溢出

 |
| **2kHz ~ 50kHz** | 计数在一周期内落在 1680 ~ 42000 之间 | **保持极高精度，CPU很轻松** | **最佳平衡点（黄金频段）** |
| **> 500kHz** | 一周期内数不了几十下，且中断极快 | ❌ 精度严重丢失，系统可能卡死

 | 采样率不足 & 中断开销过大

 |

所以，官方写这个不加分频的 Example，默认就是在测试高频且精准的小信号。你给它塞一个 100Hz 的大波浪，它小小的 16 位容器确实是装不下的。

## 🏎️ 正常硬件原理：STM32 是如何“一箭双雕”的？

你给引脚（PB7）输入了一个 PWM 信号。在标准库中，你调用了 `TIM_PWMIConfig` 函数，这个函数在 STM32 内部启动了一个非常精妙的“双路捕获 + 硬件自动复位”机制：

```
信号输入 (PB7) ──> 内部拆成两路:
                     ├──> 路A：检测上升沿 ──> 抓取数值存入 CCR2 (周期) & 顺手把计数器清零
                     └──> 路B：检测下降沿 ──> 抓取数值存入 CCR1 (高电平时间)

```

### ⏳ 一个完美周期的三步走：

假设定时器里有一个从 0 开始疯狂往上数数的计数器（CNT），每 $1\mu\text{s}$ 加 1。

1. **第一个上升沿来到**：
* 硬件触发**复位（Reset）**，把 CNT 强行清零（CNT = 0），定时器开始重新往上数数。




2. **下降沿来到（高电平结束）**：
* 硬件瞬间把当前的 CNT 值拷出来，塞进 **`CCR1`** 寄存器。此时 `CCR1` 存的就是**高电平持续的时间**。




3. **第二个上升沿来到（整个周期结束）**：
* 硬件瞬间把当前的 CNT 值拷出来，塞进 **`CCR2`** 寄存器。此时 `CCR2` 存的就是**整个周期的总时间**。


* **与此同时**，硬件再次把 CNT 清零（CNT = 0），为下一个周期做准备。





最后，CPU 只需要执行纯数学计算：

* **频率** = 定时器时钟 / `CCR2`

* **占空比** = `CCR1` / `CCR2`


---

## 💥 异常发生：100Hz 的信号进来，硬件里发生了什么？

这个硬件机制成立的核心假设是：**在“第一个上升沿”到“第二个上升沿”之间，CNT 计数器绝对不能自己数满溢出。**

一旦数满溢出，整个逻辑就崩塌了。我们来看 100Hz 信号进来时，内部发生的灾难：

```
【第一个上升沿】
   │
   ├──> CNT 被强行清零 (CNT=0)，开始以 84MHz（1秒数84,000,000下）全速往上数！
   │
   ├──> 数到 65535（仅耗时 0.78ms） ──> 【第1次硬件溢出】CNT 憋不住了，自己自动变成 0 重新数！
   ├──> 又数到 65535 ──────────────> 【第2次硬件溢出】CNT 再次变成 0 重新数！
   ├──> 又数到 65535 ──────────────> 【第3次硬件溢出】CNT 再次变成 0 重新数！
   ├──> 又数到 65535 ──────────────> 【第4次硬件溢出】CNT 再次变成 0 重新数！
   │
   ├──> 终于！过去了 6ms，【下降沿】来了！
   │    此时 CNT 刚好数到 30208。硬件把 30208 拷进 CCR1。
   │
   ├──> 又数到 65535 ──────────────> 【第5次硬件溢出】CNT 再次变成 0 重新数！
   │
   └──> 终于！过去了 10ms，【第二个上升沿】姗姗来迟！
        此时 CNT 刚好数到 6656。硬件把 6656 拷进 CCR2。

```

### 🧠 结果：CPU 被硬件给的数据骗了

当波形结束后，触发了中断，CPU 去寄存器里拿数据：

* CPU 以为 `CCR2` 里存的是整个周期的完整计数，兴高采烈地拿去算频率：

$$\text{Frequency} = \frac{84,000,000}{6656} = 12620\text{ Hz}$$


* CPU 以为 `CCR1` 和 `CCR2` 是同一条起跑线上的数据，拿去算占空比：



$$\text{DutyCycle} = \frac{30208}{6656} = 4.53 \xrightarrow{\text{C语言截断}} 4$$



但实际上，`CCR1` 经历过了 4 次溢出，`CCR2` 经历过了 5 次溢出，它们根本不在同一个时空维度了，算出来的数字自然变成了毫无逻辑的鬼画符（12606Hz，占空比4）。

---

## 🏁 总结

STM32 的 PWM 输入捕获，本质上是**依靠单次不溢出的高精度计数**来量化时间的。

* 为什么不改代码时输入 **10kHz** 能准？因为 10kHz 的周期只有 $0.1\text{ms}$，定时器全速数数也才数到 8400下，**远没有超过 65535 的容器上限**，硬件的 Reset 机制能完美闭环。


* 为什么测 **100Hz** 必须加分频器？因为分频器把定时器数数的速度降下来了，让它从“开跑车”变成“老头乐”，数了 10ms 也才数到 10000 下，同样**不会撑爆 65535 的容器**。

## 🛑 第一道关卡：软件算力崩溃线（最致命的限制）

在这份官方例程中，最严重的瓶颈不是硬件数不过来，而是 **CPU 会被中断活活累死**。

来看你发出来的中断服务函数 `TIM4_IRQHandler`：

```c
void TIM4_IRQHandler(void)
{
  RCC_ClocksTypeDef RCC_Clocks;
  RCC_GetClocksFreq(&RCC_Clocks); // 👈 罪魁祸首在这！
  ...
  DutyCycle = (TIM_GetCapture1(TIM4) * 100) / IC2Value; // 👈 还有复杂的除法[cite: 1]
  Frequency = (RCC_Clocks.HCLK_Frequency)/2 / IC2Value; // 👈 又是除法[cite: 1]
}

```

### 为什么这里会拖垮 CPU？

1. **`RCC_GetClocksFreq` 是一个极度耗时的函数**：正如你前面贴出的代码，这个函数内部要读取好几个寄存器，进行复杂的 `switch-case` 判断，还要做多次乘除法来实时计算主频。在 ARM Cortex-M4 架构中，哪怕有硬件乘除法，这个函数执行一次也要消耗 **几百个 CPU 时钟周期**。
2. **硬件除法指令的开销**：中断里还有两次 32 位的整型除法，每次消耗 2 到 12 个周期。



### 数学推导：

STM32F407 的内核主频是 **168MHz**。

* 如果信号频率是 **50kHz**，波形周期是 $20\,\mu\text{s}$。这意味着每隔 $20\,\mu\text{s}$ 就要进一次中断。
在 $20\,\mu\text{s}$ 内，CPU 总共拥有的算力是：

$$168\text{MHz} \times 20\,\mu\text{s} = 3360\text{ 个时钟周期}$$



进出中断加上执行那堆复杂的函数大约消耗 300~500 个周期。此时中断占用了 CPU 约 10%~15% 的算力，系统运行很健康。
* 如果信号频率飙升到 **500kHz**，波形周期缩短到 $2\,\mu\text{s}$。
CPU 在两次中断之间只有：

$$168\text{MHz} \times 2\,\mu\text{s} = 336\text{ 个时钟周期}$$



**灾难发生了**：CPU 刚执行完上一次中断的 `RCC_GetClocksFreq`，还没来得及退出中断，下一个波形的上升沿又触发了中断。整个主程序 `main()` 彻底瘫痪，程序直接卡死在中断里。



> **上限结论一**：受限于主库里极其臃肿的 `RCC_GetClocksFreq` 函数，为了不让 CPU 算力崩溃，信号频率**不建议超过 50kHz**。
> 
>

## 🎯 第二道关卡：数据精度失真线（量化误差）

即便把中断关了，改用 DMA 搬运数据以解放 CPU，**定时器的计数原理**也从数学上锁死了高频的精度。

定时器捕获的本质，是用 84MHz 的尺子去量度未知信号的周期。
设信号频率为 $f$，则定时器记录的总格子数（`IC2Value`）为：


$$N = \frac{84,000,000}{f}$$

占空比的计算公式是：$\text{DutyCycle} = \frac{\text{高电平格子数}}{N} \times 100\%$。

* **当信号为 10kHz 时**：

$$N = \frac{84,000,000}{10,000} = 8400\text{ 下}$$



如果占空比改变了 1%，高电平计数就会改变 84 下。此时你能测出 60.1%、60.2% 这样非常细腻的占空比。
* **当信号为 840kHz 时**：

$$N = \frac{84,000,000}{840,000} = 100\text{ 下}$$



这就意味着，整个周期定时器**总共才数了 100 下**。此时占空比的分辨率直接退化成 $\frac{1}{100} = 1\%$。你只能测出 60%、61%、62%，中间的小数点全部丢失。
* **当信号为 42MHz 时**：

$$N = \frac{84,000,000}{42,000,000} = 2\text{ 下}$$



整个周期定时器只数了 2 下。如果是 50% 占空比，高电平数 1 下，周期数 2 下；如果占空比稍微变一点点，数据直接变成 0% 或 100%。测量完全失去了意义。

> **上限结论二**：频率越高，定时器数出的格子 $N$ 就越少，测量分辨率呈指数级塌陷。为了保证占空比至少有 **0.1% 以上的分辨率**（即一个周期至少数 1000 下），信号频率**必须低于 84kHz**。

---

## ⚡ 第三道关卡：硬件物理死线（Nyquist 与同步电路）

这是 STM32 芯片内部数字电路的物理极限。

外部引脚 PB7 上的电平变化，并不是直接传给定时器的。为了防止外部噪声干扰，STM32 内部有一个**重采样同步电路**。
外部信号必须先通过定时器内部时钟（84MHz）进行同步采样。根据数字电路的设计规范，数字输入通道为了稳定锁存一个边沿，**外部信号的电平跳变沿，其脉冲宽度至少要维持 2 到 3 个定时器时钟周期**。

* 定时器时钟周期：$\frac{1}{84\text{MHz}} \approx 11.9\,\text{ns}$。
* 维持 3 个周期，意味着高电平或低电平的最短宽度不能低于：$11.9\,\text{ns} \times 3 \approx 35.7\,\text{ns}$。
* 假设在极限 50% 占空比下，一个完整周期的最短时常为：$35.7\,\text{ns} \times 2 = 71.4\,\text{ns}$。
* 换算成物理极限频率：

$$f_{\text{max}} = \frac{1}{71.4\,\text{ns}} \approx 14\text{ MHz}$$



也就是说，一旦外部信号超过 14MHz，STM32 的内部硬件甚至连边沿都捕捉不到了，直接把高频波形当成了一条平整的直线。

---

## 🏁 总结：为什么官方不分频的“黄金频段”上限是 50kHz？

把这三道关卡摆在一起，你就一目了然了：

1. **硬件物理死线**：超过 **14MHz**，硬件变瞎子，什么都看不见。
2. **数据精度线**：超过 **84kHz**，一个周期数不到 1000 下，精度开始尿崩。
3. **软件算力线**：超过 **50kHz**，因为 `RCC_GetClocksFreq` 频繁执行，CPU 开始顶不住，面临卡死风险。



所以，综合**硬件、精度、软件**三方博弈，在不加预分频、不改动官方例程公式的前提下，**50kHz** 就是绝大多数工程师公认的、能稳定跑出高精度结果的**安全上限**。
